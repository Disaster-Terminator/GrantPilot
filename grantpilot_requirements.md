# GrantPilot：ChatGPT App/Connector 批准按钮自动处理器

## 项目定位

**GrantPilot** 是一个极简浏览器插件，用来自动处理 ChatGPT 页面内 App / Connector / Tool 调用时反复出现的“批准 / 允许 / 确认”按钮。

这个项目的第一版目标非常窄：  
**只识别 ChatGPT 页面里的批准类按钮，并在用户开启插件后自动点击。**

它不是通用网页自动点击器，不是账号授权工具，也不是自动化测试框架。

## 背景

当前 ChatGPT App / Connector / MCP 工具调用场景里，经常会出现重复的人工确认按钮。用户的真实痛点不是要做一个复杂自动化系统，而是想减少这种重复确认带来的打断。

前期搜索没有发现成熟、专门面向 ChatGPT App / Connector 批准按钮的开源项目。市面上有通用 auto-clicker / auto-refresh 类浏览器扩展，但它们通常权限较宽、行为较泛，不适合直接拿来处理授权、确认、批准这类敏感按钮。

因此第一版选择做一个权限窄、行为窄、状态明确的小插件。

## 项目名

**GrantPilot**

含义：

- `Grant`：授权、批准、允许
- `Pilot`：辅助驾驶，不是完全托管
- 暗示插件只在用户明确开启后辅助处理重复批准动作

中文可叫：

**批准领航** / **授权领航**

仓库名建议：

```text
grantpilot
```

或更明确一点：

```text
chatgpt-grantpilot
```

如果未来只服务 ChatGPT，推荐 `chatgpt-grantpilot`。  
如果未来可能扩展到其他网页确认流，推荐 `grantpilot`。

## 第一版真实需求

第一版只做以下事情：

1. 浏览器插件形态。
2. 有一个浏览器工具栏图标。
3. 插件只有两个状态：
   - 开启
   - 关闭
4. 用户开启后，插件在 ChatGPT 页面里工作。
5. 插件定时检查页面，必要时可配合 DOM 变化监听。
6. 当页面出现明确的批准 / 允许 / 确认类按钮时，自动点击。
7. 用户关闭后，插件完全停止自动点击行为。
8. 第一版不需要复杂配置页。
9. 第一版不需要多站点支持。
10. 第一版不需要复杂规则引擎。

## 作用范围

第一版作用范围应限制在：

```text
https://chatgpt.com/*
```

可根据实际页面情况兼容：

```text
https://chat.openai.com/*
```

但默认目标是新版 ChatGPT 页面。

## 明确不做

第一版不要做以下事情：

- 不做通用网页自动点击器。
- 不支持任意网站。
- 不自动点击第三方 OAuth 页面。
- 不自动点击 Google / GitHub / Microsoft / Notion / Slack 等外部授权页。
- 不自动处理登录。
- 不自动创建连接器。
- 不自动安装 App。
- 不自动选择账号。
- 不自动授予新增外部权限。
- 不点击支付、删除、组织授权、管理员授权、仓库授权等高风险按钮。
- 不做云端同步。
- 不收集 cookie、token、账号信息、聊天内容。
- 不做复杂规则市场。
- 不做 AI 识别按钮。
- 不做 OCR。
- 不做 Playwright 后台浏览器。
- 不做桌面端自动化。
- 不做多 agent 编排。

## 按钮识别口径

插件需要识别的是 ChatGPT 页面内部出现的批准类按钮，而不是所有看起来像“允许”的按钮。

候选按钮可以包括：

```text
button
[role="button"]
```

按钮文本或可访问性文本可能包括：

```text
Allow
Approve
Confirm
Continue
允许
批准
确认
继续
```

但不能只靠按钮文本。  
为了避免误点，按钮附近上下文也应该与 ChatGPT App / Connector / Tool 调用相关。

附近上下文可以包含类似含义：

```text
app
connector
tool
MCP
action
应用
连接器
工具
工具调用
操作
```

禁止点击的按钮文本包括：

```text
Deny
Cancel
Reject
拒绝
取消
```

第一版可以先以实际页面 DOM 为准，做最小可用匹配；但目标仍然是“只处理 ChatGPT 内部的批准确认”。

## 用户可见行为

用户看到的插件应该非常简单：

### 关闭状态

- 图标显示关闭状态。
- 插件不扫描、不点击。
- 不影响 ChatGPT 页面。

### 开启状态

- 图标显示开启状态。
- 插件开始在 ChatGPT 页面内检查批准按钮。
- 发现按钮后自动点击。
- 点击后可记录最近一次动作，方便排查。

### 可选但推荐的最小信息

插件弹窗里可以显示：

```text
Status: Enabled / Disabled
Last action: Never / 2026-05-06 17:30
Last matched button: Allow
```

不需要做复杂日志页面。

## 刷新与检查行为

用户原始需求里提到“选开就定时刷新，然后看到批准就点”。

这里的“定时刷新”可以理解为：

- 定时扫描当前页面 DOM；
- 不一定要真的刷新整个网页。

第一版更适合做成：

1. 开启后周期性扫描 DOM。
2. 如果页面 DOM 变化，也触发一次检查。
3. 找到明确目标按钮后点击。
4. 避免高频重复点击。

不要求第一版真的调用浏览器刷新页面。  
如果后续确认 ChatGPT 页面必须刷新才能出现按钮，再单独加“页面刷新”选项。

## 状态设计

第一版只有两个主状态：

```text
enabled = true
enabled = false
```

状态需要持久化，浏览器重启后仍能保留用户选择。

可以使用浏览器扩展本地存储保存：

```text
enabled
lastActionAt
lastMatchedText
```

## 安全边界

这是一个会自动点击批准按钮的工具，所以边界必须清楚。

第一版的安全原则：

1. 默认关闭。
2. 只在用户手动开启后工作。
3. 只在 ChatGPT 域名下工作。
4. 不碰外部 OAuth / 登录 / 支付 / 权限升级页面。
5. 不读取、不上传、不保存聊天内容。
6. 不收集账号凭据。
7. 不做远程规则更新。
8. 不做隐藏行为。
9. 用户可以随时一键关闭。

## 验收标准

第一版可认为完成，当它满足：

1. 插件可以在 Chrome / Edge 开发者模式加载。
2. 工具栏图标可打开一个小弹窗。
3. 弹窗可以切换 Enabled / Disabled。
4. Disabled 状态下不会点击任何按钮。
5. Enabled 状态下，在 ChatGPT 页面内能扫描候选批准按钮。
6. 命中批准按钮后能自动点击。
7. 点击后不会进入无限重复点击。
8. 插件权限只覆盖必要域名，不使用 `<all_urls>`。
9. 不访问外部授权页。
10. 没有后台网络请求。
11. 代码和 README 能清楚说明第一版边界。

## README 应该表达的核心内容

README 不需要写得像大产品。  
它只需要说明：

- 这个插件解决什么痛点；
- 它只在 ChatGPT 页面里工作；
- 它需要用户手动开启；
- 它会自动点击批准类按钮；
- 它不会处理外部 OAuth 授权；
- 它不收集数据；
- 当前版本是 MVP，行为刻意很窄。

## 给实现 agent 的一句话任务

实现一个名为 **GrantPilot** 的极简 Chrome/Edge MV3 浏览器插件：用户通过工具栏图标切换开启/关闭；开启后只在 ChatGPT 页面内周期性检查并自动点击 App / Connector / Tool 调用相关的批准/允许/确认按钮；关闭后完全不动作。第一版不要做通用自动点击器、外部 OAuth 授权处理、复杂配置、云同步或任何无关功能。
